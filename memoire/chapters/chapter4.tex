\section{Introduction}
% ────────────────────────────────────────────────────────────

Ce chapitre présente en détail la contribution principale de ce travail : la conception et l'implémentation complète de \textbf{ShieldNet}, un système distribué de mitigation collaborative des attaques par déni de service distribué (DDoS).

ShieldNet repose sur une architecture pair-à-pair (P2P) dans laquelle des nœuds hétérogènes — universités, fournisseurs d'accès Internet, centres de données, PME — forment une coalition de défense mutualisée. Chaque nœud conserve son autonomie décisionnelle tout en participant à un mécanisme de solidarité : lorsqu'un nœud est submergé par un volume d'attaque dépassant sa capacité locale, il peut solliciter l'aide de ses pairs de confiance pour absorber le surplus de trafic malveillant.

L'implémentation couvre sept modules interdépendants :

\begin{enumerate}
    \item Le modèle de données relationnel (14 tables PostgreSQL)
    \item Le module d'authentification asymétrique JWT RS256 et mTLS
    \item Le module de découverte et de gestion des pairs
    \item Le module de coalition anti-DDoS
    \item L'algorithme de sélection WSM (\textit{Weighted Sum Model})
    \item Le modèle de confiance PeerTrust
    \item L'infrastructure de déploiement Docker et le script de démonstration
\end{enumerate}

\subsection{Table PEERS}

La table \texttt{PEERS} est centrale dans l'architecture. Elle stocke le répertoire de tous les pairs connus par un nœud donné. Chaque entrée contient la clé publique PEM du pair, utilisée pour la vérification des tokens JWT.

\begin{lstlisting}[language=JavaScript, caption={Modele Sequelize de la table PEERS (extrait)}, label={lst:peer-model}]
peer_id:    { type: UUID, primaryKey: true, defaultValue: UUIDV4 },
peer_name:  { type: STRING, allowNull: false },
organization_type: { type: ENUM('UNIVERSITY','ISP','DATACENTER','PME',...) },
tier:       { type: ENUM('T1','T2','T3') },
public_key: { type: TEXT, allowNull: false },  // Cle publique PEM
declared_available_gbps: { type: FLOAT },      // Capacite disponible
measured_latency_ms:     { type: FLOAT },      // Latence mesuree (RTT)
status: { type: ENUM('ACTIVE','INACTIVE','MAINTENANCE','SUSPECTED','BANNED') },
membership_status: { type: ENUM('PROBATION','CONFIRMED','SUSPENDED','EXPELLED') },
consecutive_missed_heartbeats: { type: INTEGER, defaultValue: 0 },
\end{lstlisting}

Les champs \texttt{declared\_available\_gbps} et \texttt{measured\_latency\_ms} sont mis à jour à chaque heartbeat reçu. Ils constituent les entrées directes de l'algorithme WSM de sélection des pairs (section~\ref{sec:wsm}).

\subsection{Initialisation et synchronisation de la base}

Lors du premier démarrage, Sequelize synchronise automatiquement le schéma via \texttt{sequelize.sync(\{alter: false\})}. Un mécanisme de garde capture l'erreur PostgreSQL \texttt{42P07} (relation déjà existante) pour éviter les recréations de tables lors des redémarrages.

\begin{lstlisting}[language=JavaScript, caption={Initialisation securisee de la base de donnees}, label={lst:db-init}]
async function initDatabase() {
  await sequelize.authenticate();
  try {
    await sequelize.sync({ alter: false });
  } catch (err) {
    // 42P07 = table/index already exists -- schema already initialized
    if (err.parent?.code !== '42P07') throw err;
  }
  return db;
}
\end{lstlisting}

\section{Module d'Authentification : JWT RS256 et mTLS}
\label{sec:auth}
% ────────────────────────────────────────────────────────────

\subsection{Principes de sécurité}

La sécurité des communications entre nœuds ShieldNet repose sur deux mécanismes complémentaires :

\begin{enumerate}
    \item \textbf{JWT RS256} : authentification asymétrique à courte durée de vie (60 secondes)
    \item \textbf{mTLS} : authentification mutuelle par certificat X.509 au niveau transport
\end{enumerate}

L'utilisation de l'algorithme \textbf{RS256} (RSA + SHA-256) est fondamentale : contrairement à HS256 (secret partagé), chaque nœud possède sa propre paire de clés RSA. Un nœud compromis n'expose donc pas les autres membres de la coalition. Le standard JWT est défini dans la RFC~7519~\cite{rfc7519jwt} et les algorithmes cryptographiques dans la RFC~7518~\cite{rfc7518jwa}.

\subsection{Génération du token}

La fonction \texttt{generateToken()} lit la clé privée RSA du nœud local depuis le système de fichiers, puis signe un payload minimal avec l'algorithme RS256 :

\begin{lstlisting}[language=JavaScript, caption={Generation d'un token JWT RS256}, label={lst:jwt-gen}]
function generateToken(node_id) {
  const privateKey = fs.readFileSync(process.env.TLS_KEY || './certs/node.key');
  return jwt.sign(
    { iss: node_id, node_id },
    privateKey,
    { algorithm: 'RS256', expiresIn: '60s' }
  );
}
\end{lstlisting}

Le claim \texttt{iss} (issuer) contient l'identifiant UUID du nœud émetteur. Le récepteur utilise ce claim pour récupérer la clé publique correspondante dans sa table \texttt{PEERS}.

\subsection{Vérification du token (middleware)}

Le middleware \texttt{jwtMiddleware} intercepte toutes les requêtes entrantes (à l'exception des routes publiques) et effectue la vérification en trois étapes :

\begin{lstlisting}[language=JavaScript, caption={Middleware de verification JWT RS256}, label={lst:jwt-verify}]
async function jwtMiddleware(req, res, next) {
  if (isPublicRoute(req)) return next();

  const token = extractToken(req);  // Bearer header, body ou query

  // Etape 1 : decoder sans verifier pour lire l'emetteur
  const decoded = jwt.decode(token);
  const issuer_id = decoded.iss;

  // Etape 2 : recuperer la cle publique de l'emetteur
  let publicKey;
  const isLocal = process.env.NODE_ID === issuer_id ||
    (localNode && localNode.node_id === issuer_id);

  if (isLocal) {
    publicKey = fs.readFileSync(process.env.TLS_CERT || './certs/node.crt');
  } else {
    const peer = await Peer.findOne({ where: { peer_id: issuer_id } });
    if (peer.status === 'BANNED') return res.status(403)...;
    publicKey = peer.public_key;  // PEM stocke en base
  }

  // Etape 3 : verifier la signature RS256
  jwt.verify(token, publicKey, { algorithms: ['RS256'] });
  req.authNode = authInfo;
  next();
}
\end{lstlisting}

\subsection{Correction d'un bug critique de bootstrapping}

Lors de l'implémentation, un bug subtil a été identifié : la vérification de l'identité locale utilisait un court-circuit logique qui échouait silencieusement si la base de données était vide (premier démarrage).

\begin{lstlisting}[language=JavaScript, caption={Bug original -- court-circuit sur localNode null}, label={lst:bug}]
// Bug : si localNode est null (DB vide), isLocalNode = false
// meme si process.env.NODE_ID correspond !
const isLocalNode = localNode &&
  (localNode.node_id === node_id || localNode.node_name === node_id);
\end{lstlisting}

\begin{lstlisting}[language=JavaScript, caption={Correction -- verification de l'env en premier}, label={lst:fix}]
// Correction : verifier process.env.NODE_ID avant de consulter la DB
const isLocalNode =
  process.env.NODE_ID   === node_id ||
  process.env.NODE_NAME === node_id ||
  (localNode && (localNode.node_id === node_id ||
                 localNode.node_name === node_id));

const effectiveNodeId =
  (localNode && localNode.node_id) || process.env.NODE_ID || node_id;
\end{lstlisting}

Ce bug provoquait une erreur HTTP 401 «~Unknown node~» lors du tout premier appel à \texttt{/auth/token}, rendant le système impossible à initialiser. La même correction a été appliquée symétriquement dans le middleware de vérification.

\subsection{Middleware mTLS}

En complément du JWT, le middleware \texttt{mtlsMiddleware} lit le certificat client présenté lors de la connexion TLS (protocole TLS~1.3 défini dans la RFC~8446~\cite{rfc8446tls13}) et l'attache à la requête pour usage ultérieur :

\begin{lstlisting}[language=JavaScript, caption={Middleware mTLS}, label={lst:mtls}]
function mtlsMiddleware(req, res, next) {
  if (process.env.MTLS_ENABLED !== 'true') return next();

  const cert = req.socket.getPeerCertificate();
  if (!cert || !cert.subject) {
    if (process.env.MTLS_STRICT === 'true')
      return res.status(401).json({ error: 'mTLS required' });
    return next(); // Mode non-strict : continuer sans cert
  }

  req.clientCert = {
    subject: cert.subject, issuer: cert.issuer,
    fingerprint: cert.fingerprint,
    valid_from: cert.valid_from, valid_to: cert.valid_to,
  };
  next();
}
\end{lstlisting}

Le mode non-strict (\texttt{MTLS\_STRICT=false}) permet de conserver la compatibilité avec des clients qui ne présentent pas de certificat, tout en enrichissant la requête lorsque le certificat est présent.

\subsection{Enregistrement transactionnel des pairs}

L'enregistrement d'un pair est implémenté dans une transaction Sequelize pour garantir l'atomicité de l'opération (pair + capacités associées) :

\begin{lstlisting}[language=JavaScript, caption={Enregistrement transactionnel d'un pair}, label={lst:peer-register}]
const peerId = await sequelize.transaction(async (transaction) => {
  let peer = peerPayload.peer_id
    ? await Peer.findByPk(peerPayload.peer_id, { transaction })
    : null;

  if (peer) {
    await peer.update(payload, { transaction });
  } else {
    created = true;
    peer = await Peer.create({ ...payload }, { transaction });
  }

  // Enregistrement ou mise a jour de la capacite declaree du pair
  for (const capability of capabilities) {
    const existing = await PeerCapability.findOne({
      where: { peer_id: peer.peer_id }, transaction,
    });
    const values = {
      peer_id: peer.peer_id,
      declared_capacity_gbps: capability.declared_capacity_gbps
                              ?? capability.capacity_gbps ?? 0,
      last_updated: new Date(),
    };
    if (existing) await existing.update(values, { transaction });
    else          await PeerCapability.create(values, { transaction });
  }
  return peer.peer_id;
});

// Creation automatique du ledger de reciprocite
if (created) {
  await ReciprocityLedger.findOrCreate({
    where: { peer_id: peerId },
    defaults: { peer_id: peerId, credits_received: 0, credits_given: 0 },
  });
}
\end{lstlisting}

\subsection{Protocole de heartbeat}

Le mécanisme de heartbeat maintient la connaissance de la disponibilité des pairs. À chaque réception, le nœud met à jour la capacité disponible et la latence mesurée du pair émetteur :

\begin{lstlisting}[language=JavaScript, caption={Traitement d'un heartbeat recu}, label={lst:heartbeat-receive}]
// Mise a jour du pair avec les nouvelles metriques
await peer.update({
  last_heartbeat:         new Date(),
  declared_available_gbps: reported_available_gbps,
  measured_latency_ms:    round_trip_time_ms ?? peer.measured_latency_ms,
  consecutive_missed_heartbeats: 0,
  status: peer.status === 'BANNED' ? 'BANNED' : 'ACTIVE',
});
\end{lstlisting}

La règle «~si BANNED, rester BANNED~» assure qu'un pair sanctionné ne peut pas se réhabiliter simplement en envoyant un heartbeat.

Les endpoints de négociation gèrent les transitions d'état de la session :

\subsection{Clôture et mise à jour des crédits}

La clôture d'une attaque (\texttt{POST /attack/over}) est l'étape la plus critique : elle consolide les données qui serviront au calcul du score de confiance.

\begin{lstlisting}[language=JavaScript, caption={Cloture d'une session et mise a jour du ledger de reciprocite}, label={lst:attack-over}]
for (const session of sessions) {
  // Si actual_volume n'a pas ete renseigne lors de la redirection,
  // on considere que le pair a fourni exactement ce qu'il avait promis
  if (!session.actual_volume_gbps) {
    await session.update({
      actual_volume_gbps: session.accepted_volume_gbps
                          ?? session.requested_volume_gbps ?? 0,
    });
  }

  const volume = session.actual_volume_gbps || 0;

  // Mise a jour du ledger de reciprocite
  const [ledger] = await ReciprocityLedger.findOrCreate({
    where: { peer_id: session.helping_peer_id },
    defaults: { credits_received: 0, credits_given: 0, balance: 0 },
  });
  await ledger.update({
    credits_given: ledger.credits_given + volume,
    balance:       ledger.balance + volume,
  });
}
\end{lstlisting}

\textbf{Note importante :} Le champ \texttt{actual\_volume\_gbps} peut être \texttt{NULL} si l'endpoint \texttt{/traffic/redirect} n'est pas explicitement appelé. Le garde \texttt{if (!session.actual\_volume\_gbps)} assure qu'un volume nul n'entraîne pas un score $S_k = 0$ (ce qui produirait un BANNED injustifié).

\section{Algorithme WSM de Sélection des Pairs}
\label{sec:wsm}
% ────────────────────────────────────────────────────────────

\subsection{Formalisation mathématique}

L'algorithme de sélection des pairs est fondé sur le \textbf{Weighted Sum Model} (WSM)~\cite{hwang1981wsm}, une méthode classique d'aide à la décision multi-critères. Pour un pair $p$, le score composite est calculé comme suit :

\begin{equation}
\text{Score}(p) = w_C \cdot C_p + w_L \cdot L_p' + w_T \cdot T_p + w_R \cdot R_p
\label{eq:wsm}
\end{equation}

où les critères normalisés sont définis par :

\begin{align}
C_p  &= \frac{\text{cap\_disp}_p}{\max_j(\text{cap\_disp}_j)}
    & &\text{(capacité disponible normalisée)} \\
L_p' &= 1 - \frac{L_p - L_{\min}}{L_{\max} - L_{\min}}
    & &\text{(latence inversée normalisée)} \\
T_p  &= \text{overall\_score}(p)
    & &\text{(score de confiance PeerTrust)} \\
R_p  &= \frac{\text{credits\_received}_p}{\text{credits\_received}_p + \text{credits\_given}_p + \varepsilon}
    & &\text{(réciprocité bilatérale)}
\end{align}

\subsection{Poids AHP validés}

Les poids $w_C$, $w_L$, $w_T$, $w_R$ ont été déterminés par la méthode AHP (\textit{Analytic Hierarchy Process}) introduite par Saaty~\cite{saaty1980ahp}. La cohérence du jugement est mesurée par le ratio de cohérence CR = 1,6\,\% $< 10\,\%$ (seuil d'acceptabilité fixé à 10\,\%~\cite{saaty1980ahp}), validant la fiabilité de la pondération.

\begin{table}[H]
\centering
\caption{Poids AHP pour l'algorithme WSM}
\label{tab:ahp-weights}
\begin{tabular}{|l|c|c|}
\hline
\textbf{Critère} & \textbf{Variable} & \textbf{Poids} \\
\hline
Capacité disponible  & $w_C$ & 0,52 \\
Latence (inversée)   & $w_L$ & 0,20 \\
Confiance PeerTrust  & $w_T$ & 0,20 \\
Réciprocité          & $w_R$ & 0,08 \\
\hline
\textbf{Total}       &       & \textbf{1,00} \\
\hline
\end{tabular}
\end{table}

La capacité disponible reçoit le poids le plus élevé (0,52) car, en situation de crise DDoS, la capacité effective d'absorption est le facteur limitant primordial.

\subsection{Implémentation}

\begin{lstlisting}[language=JavaScript, caption={Calcul du score WSM et normalisation des criteres}, label={lst:wsm-impl}]
const W = { C: 0.52, L: 0.20, T: 0.20, R: 0.08 };
const EPSILON = 1e-6;

// Normalisation
const maxCap  = Math.max(...capValues);
const latRange = maxLat - minLat;

const scored = candidates.map((peer, i) => {
  const Cp = maxCap > 0 ? capValues[i] / maxCap : 0;

  // Latence inversee : plus faible latence = score plus eleve
  const Lp_inv = latRange > 0
    ? 1 - (latValues[i] - minLat) / latRange
    : 1;  // si tous identiques, score maximal pour tous

  const Tp = peer.trust_score?.overall_score ?? 0.5;

  const received = Number(peer.reciprocity_ledger?.credits_received ?? 0);
  const given    = Number(peer.reciprocity_ledger?.credits_given    ?? 0);
  const Rp = received / (received + given + EPSILON);

  const score = W.C * Cp + W.L * Lp_inv + W.T * Tp + W.R * Rp;
  return { peer, score, criteria: { Cp, Lp_inv, Tp, Rp } };
});
\end{lstlisting}

\subsection{Répartition proportionnelle}

Tous les pairs éligibles participent à l'absorption du trafic. Le score WSM de chaque pair est transformé en poids de répartition, de sorte que la somme des pourcentages attribués vaille toujours 100\,\% :

\begin{equation}
w_i = \frac{\text{Score}(p_i)}{\displaystyle\sum_{j=1}^{N} \text{Score}(p_j)}
\label{eq:wsm-weight}
\end{equation}

\begin{equation}
\text{allocation\%}(p_i) = w_i \times 100
\label{eq:alloc-pct}
\end{equation}

Cette formulation présente deux avantages essentiels. D'une part, le volume total de trafic à redistribuer n'est pas requis en entrée : la répartition est exprimée en pourcentages, valables quelle que soit l'intensité de l'attaque. D'autre part, tous les pairs contribuent à l'effort collectif selon leur valeur relative, ce qui est conforme à l'esprit de la coalition de défense mutualisée.

\begin{lstlisting}[language=JavaScript, caption={Repartition proportionnelle par poids WSM normalises}, label={lst:wsm-alloc}]
// Somme totale des scores WSM
const totalScore = scored.reduce((sum, c) => sum + c.score, 0);

// Pour chaque pair : poids et pourcentage du flux
const plan = scored.map((candidate) => {
  // w_i = Score(p_i) / somme Score(p_j)
  const weight = totalScore > 0
    ? candidate.score / totalScore
    : 1 / scored.length;  // repartition egale si scores tous nuls

  const allocation_pct = Number((weight * 100).toFixed(2));

  return {
    peer:           { /* informations du pair */ },
    wsm_score:      Number(candidate.score.toFixed(4)),
    weight:         Number(weight.toFixed(4)),
    allocation_pct,
  };
});
\end{lstlisting}

Considérons un scénario avec 3 pairs. Le volume excédentaire est inconnu a priori ; la répartition est calculée en pourcentages :

\section{Modèle de Confiance PeerTrust}
\label{sec:trust}
% ────────────────────────────────────────────────────────────

\subsection{Formulation mathématique}

Le modèle de confiance PeerTrust a été proposé par Xiong et Liu~\cite{xiong2004peertrust} dans le cadre des communautés électroniques P2P. Il définit plusieurs variantes de la métrique de confiance. Après analyse des contraintes spécifiques de ShieldNet — coalition fermée de 4 nœuds, transactions hétérogènes en sévérité, nécessité de pondérer différemment les sessions critiques — nous avons retenu l'\textbf{équation 6} du papier original~\cite{xiong2004peertrust}, dite \emph{métrique avec facteur de contexte transactionnel} :

\begin{equation}
T(u) = \sum_{i=1}^{I(u)} S(u,i) \cdot \mathrm{Cr}\!\left(p(u,i)\right) \cdot D(u,i)
\label{eq:peertrust}
\end{equation}

Les trois composantes sont (d'après~\cite{xiong2004peertrust}) :

\begin{align}
S(u,i)   &= \min\!\left(1,\; \frac{V_{\text{réel},i}}{V_{\text{accepté},i}}\right)
    \quad \text{(satisfaction normalisée de la session } i\text{)} \label{eq:sat}\\
\mathrm{Cr}\!\left(p(u,i)\right) &= T\!\left(\text{requesting\_peer}_i\right)\ \text{ou}\ 0{,}5\ \text{si inconnu}
    \quad \text{(crédibilité de l'évaluateur)} \label{eq:cred}\\
D(u,i)   &= \text{severity\_weight}_i \in \{0{,}25;\; 0{,}50;\; 0{,}75;\; 1{,}00\}
    \quad \text{(contexte de la transaction)} \label{eq:ctx}
\end{align}

\paragraph{Justification du choix.}
L'équation 6 a été préférée aux autres variantes du papier pour les raisons suivantes :
\begin{itemize}
    \item \textbf{Facteur de contexte $D(u,i)$} : une aide lors d'une attaque CRITICAL (1{,}00) a davantage de valeur informationnelle qu'une aide lors d'une attaque LOW (0{,}25). Sans ce facteur, toutes les sessions contribuent également au score, indépendamment de leur gravité.
    \item \textbf{Crédibilité dynamique $\mathrm{Cr}(p)$} : le score de confiance du pair requérant sert de crédibilité — un pair fiable (GOLD/SILVER) voit son évaluation pondérée plus fortement qu'un pair peu fiable.
    \item \textbf{PSM inadapté} : la variante \emph{Personalized Similarity Metric} (PSM) requiert un grand nombre de pairs communs entre évaluateurs~\cite{xiong2004peertrust}, condition non satisfaite dans une coalition de 4 nœuds.
    \item \textbf{TVM inadapté} : la variante TVM ne tient pas compte du contexte transactionnel — toutes les sessions ont le même poids quelle que soit la sévérité de l'attaque.
\end{itemize}

\subsection{Facteur de contexte transactionnel $D(u,i)$}

Le facteur $D(u,i)$ quantifie l'importance de la transaction $i$ selon la sévérité de l'attaque associée. Une aide fournie lors d'une attaque CRITICAL contribue quatre fois plus au score qu'une aide lors d'une attaque LOW :

\begin{table}[H]
\centering
\caption{Facteur de contexte $D(u,i)$ selon la sévérité de l'attaque}
\label{tab:severity-weights}
\begin{tabular}{|l|c|l|}
\hline
\textbf{Sévérité} & \textbf{$D(u,i)$} & \textbf{Interprétation} \\
\hline
LOW      & 0,25 & Attaque mineure, impact limité \\
MEDIUM   & 0,50 & Attaque modérée \\
HIGH     & 0,75 & Attaque sévère, service dégradé \\
CRITICAL & 1,00 & Attaque majeure, service hors ligne \\
\hline
\end{tabular}
\end{table}

\subsection{Niveaux de confiance}

Le score continu $T(p) \in [0, 1]$ est discrétisé en cinq niveaux :

\begin{table}[H]
\centering
\caption{Niveaux de confiance et seuils associés}
\label{tab:trust-levels}
\begin{tabular}{|l|c|l|}
\hline
\textbf{Niveau} & \textbf{Seuil} & \textbf{Signification} \\
\hline
GOLD    & $\geq 0{,}80$ & Pair hautement fiable \\
SILVER  & $\geq 0{,}60$ & Pair fiable \\
BRONZE  & $\geq 0{,}40$ & Pair acceptable \\
SUSPECT & $\geq 0{,}20$ & Pair peu fiable, historique dégradé \\
BANNED  & $< 0{,}20$    & Pair banni automatiquement \\
\hline
\end{tabular}
\end{table}

\subsection{Implémentation du calcul}

\begin{lstlisting}[language=JavaScript, caption={Calcul du score PeerTrust -- Eq. 6 avec credibilite reseau (PeerTrust)}, label={lst:trust-calc}]
async function computeTrustScore(peer_id) {
  const [sessions, localNode] = await Promise.all([
    HelpSession.findAll({
      where: { helping_peer_id: peer_id, status: 'COMPLETED' },
      include: [{ model: Attack, as: 'attack', attributes: ['severity'] }],
    }),
    LocalNodeConfig.findOne(),
  ]);

  if (sessions.length === 0) {
    return { score: 0.5, level: 'BRONZE', session_count: 0 }; // cold start
  }

  // Cr(p) = T(noeud_local) selon le reseau (mecanisme PeerTrust)
  // Interroge TOUS les pairs actifs (sauf le pair evalue) :
  //   "Quel est votre score de confiance pour moi ?"
  // Cr = moyenne des reponses --> opinion collective du reseau
  let Cr = 0.5; // valeur par defaut (cold start)
  if (localNode) {
    Cr = await fetchCrFromNetwork(localNode.node_id, peer_id);
  }

  // T(u) = somme S(u,i) * Cr(p(u,i)) * D(u,i)  -- Eq. 6 originale
  let score = 0;

  for (const session of sessions) {
    const accepted = Number(session.accepted_volume_gbps ?? 0);
    const actual   = Number(session.actual_volume_gbps   ?? 0);

    // S(u,i) = min(1, V_reel / V_accepte)
    const S = accepted > 0 ? Math.min(1, actual / accepted) : 0;
    // D(u,i) = facteur contextuel selon la severite
    const D = SEVERITY_WEIGHTS[session.attack?.severity ?? 'LOW'];

    score += S * Cr * D;
  }

  score = Math.min(1, Math.max(0, score));

  return { score, level: scoreToLevel(score), session_count: sessions.length };
}
\end{lstlisting}

\subsection{Persistance et actions automatiques}

La fonction \texttt{recalculateAndSave()} calcule le score, le persiste dans \texttt{TRUST\_SCORES}, et déclenche automatiquement le bannissement si le score tombe en dessous de 0,20 :

\begin{lstlisting}[language=JavaScript, caption={Persistance et bannissement automatique}, label={lst:trust-save}]
async function recalculateAndSave(peer_id) {
  const { score, level } = await computeTrustScore(peer_id);

  const [trustScore, created] = await TrustScore.findOrCreate({
    where: { peer_id },
    defaults: { overall_score: score, trust_level: level },
  });

  if (!created) {
    const previousLevel = trustScore.trust_level;
    await trustScore.update({ overall_score: score, trust_level: level });

    // Log si le niveau a change
    if (previousLevel !== level) {
      logAudit({ event_type: 'TRUST_LEVEL_CHANGED',
        description: `${previousLevel} -> ${level} (score=${score.toFixed(3)})` });
    }
  }

  // Bannissement automatique si score < 0.20
  if (level === 'BANNED') {
    await peer.update({
      status: 'BANNED',
      membership_status: 'EXPELLED'
    });
    logAudit({ event_type: 'PEER_BANNED', severity: 'CRITICAL' });
  }
}
\end{lstlisting}

\subsection{Exemple de dégradation du score}

Considérons un pair avec trois sessions d'aide où il ne livre pas le volume promis ($\mathrm{Cr} = 0{,}5$ par défaut) :

\begin{align*}
\text{Session 1} &: S_1 = \frac{0}{10} = 0,\quad D_1 = 1{,}00\ (\text{CRITICAL}) \\
\text{Session 2} &: S_2 = \frac{1}{10} = 0{,}10,\quad D_2 = 0{,}75\ (\text{HIGH}) \\
\text{Session 3} &: S_3 = \frac{1}{10} = 0{,}10,\quad D_3 = 0{,}50\ (\text{MEDIUM}) \\[6pt]
T(p) &= \sum_{i=1}^{3} S_i \cdot \mathrm{Cr}_i \cdot D_i \\
     &= (0 \times 0{,}5 \times 1{,}00)
      + (0{,}10 \times 0{,}5 \times 0{,}75)
      + (0{,}10 \times 0{,}5 \times 0{,}50) \\
     &= 0 + 0{,}0375 + 0{,}025 = 0{,}0625
\end{align*}

Ce score de 0,0625 est inférieur à 0,20 → le pair est automatiquement \textbf{BANNED}.

\subsection{Architecture du service}

Le service de heartbeat (\texttt{utils/heartbeatSender.js}) s'exécute en tâche de fond sur chaque nœud. Il envoie périodiquement l'état du nœud local à tous ses pairs connus, permettant de maintenir une vue cohérente de la disponibilité de la coalition.

\begin{lstlisting}[language=JavaScript, caption={Cycle d'envoi des heartbeats}, label={lst:heartbeat-cycle}]
async function sendHeartbeats() {
  const localNode = await LocalNodeConfig.findOne();
  if (!localNode) {
    console.warn('[Heartbeat] Noeud non initialise -- ignore');
    return;
  }

  // JWT frais (60s) pour chaque cycle
  const token = generateToken(localNode.node_id);

  // Pairs actifs ou inactifs (pas les bannis ni les suspectés)
  const peers = await Peer.findAll({
    where: { status: ['ACTIVE', 'INACTIVE'] },
  });

  // Envoi en parallele a tous les pairs
  await Promise.allSettled(
    peers.map((peer) => sendHeartbeatToPeer(peer, localNode, token))
  );
}

function start() {
  // Delai initial de 10s pour laisser les autres noeuds demarrer
  setTimeout(() => {
    sendHeartbeats();
    _timer = setInterval(sendHeartbeats, HEARTBEAT_INTERVAL_MS); // 30s
  }, 10_000);
}
\end{lstlisting}

  await peer.update(update);
}

    -subj "/CN=shieldnet-university/O=ShieldNet/C=DZ"
\end{lstlisting}

La clé publique (\texttt{node.crt}) est distribuée aux autres nœuds lors de l'enregistrement via \texttt{POST /peers/register}. Elle est stockée dans le champ \texttt{public\_key} de la table \texttt{PEERS} et utilisée pour la vérification des tokens JWT.

\subsection{Client HTTPS/mTLS}

Le script \texttt{scripts/mtls-request.js} est un client HTTPS léger utilisé depuis PowerShell pour effectuer des appels sécurisés lors de la démonstration. Il résout un problème critique de sérialisation :

\begin{lstlisting}[language=JavaScript, caption={Client mTLS -- resolution du chemin de certificat PEM}, label={lst:mtls-client}]
// Resoudre public_key_file -> contenu PEM
// (evite de passer le PEM en base64 via PowerShell 5.1 ConvertTo-Json
//  qui se bloque sur les longues chaines base64)
if (payload.body && payload.body.public_key_file) {
  payload.body.public_key = fs.readFileSync(
    payload.body.public_key_file, 'utf8'
  );
  delete payload.body.public_key_file;
}
\end{lstlisting}

\textbf{Problème résolu :} PowerShell 5.1 possède un bug connu dans \texttt{ConvertTo-Json} : il se bloque indéfiniment lors de la sérialisation de longues chaînes base64 (comme les certificats PEM). La solution adoptée consiste à passer uniquement le \emph{chemin} du fichier de certificat via PowerShell, puis à faire lire son contenu par Node.js côté client.

\subsection{Mesures de sécurité implémentées}

\begin{enumerate}
    \item \textbf{Tokens courts (60s)} : limite la fenêtre de rejeu en cas d'interception
    \item \textbf{Isolation des clés} : chaque nœud possède sa propre paire RSA — aucun secret partagé
    \item \textbf{Pair banni irréversible} : un pair BANNED ne peut pas se réhabiliter via un heartbeat
    \item \textbf{Fichiers sensibles exclus de git} : \texttt{.gitignore} exclut \texttt{*.key}, \texttt{*.pem}, \texttt{.env.*}
    \item \textbf{Validation des entrées} : tous les endpoints vérifient les champs obligatoires
    \item \textbf{Transactions atomiques} : l'enregistrement des pairs utilise des transactions Sequelize
    \item \textbf{Helmet.js} : en-têtes HTTP de sécurité activés sur toutes les réponses
\end{enumerate}

\section{Correction d'un Bug Critique dans le Calcul de Crédibilité}
\label{sec:bug-probation}
% ────────────────────────────────────────────────────────────

Lors de l'implémentation du mécanisme \texttt{fetchCrFromNetwork()} dans \texttt{utils/trustManager.js}, un bug subtil a provoqué une erreur PostgreSQL bloquante. La requête Sequelize filtrait les pairs par \texttt{status: ['ACTIVE', 'PROBATION']}, mais \texttt{PROBATION} appartient à l'énumération \texttt{membership\_status}, non à \texttt{status} :

\begin{lstlisting}[language=JavaScript, caption={Bug original -- valeur PROBATION invalide pour le champ status}, label={lst:probation-bug}]
// Bug : PROBATION n'est pas une valeur valide de l'enum status
// -> PostgreSQL leve : invalid input value for enum "enum_PEERS_status": "PROBATION"
where: { status: ['ACTIVE', 'PROBATION'] }
\end{lstlisting}

\begin{lstlisting}[language=JavaScript, caption={Correction -- utilisation des valeurs correctes de l'enum status}, label={lst:probation-fix}]
// Correction : seules les valeurs de PEER_STATUSES sont valides pour status
// PEER_STATUSES = ['ACTIVE', 'INACTIVE', 'MAINTENANCE', 'SUSPECTED', 'BANNED']
where: { status: ['ACTIVE', 'INACTIVE'] }
\end{lstlisting}

Ce bug provoquait un HTTP 500 lors de tout recalcul du score PeerTrust (Phase 6 du test fonctionnel), rendant les assertions sur les scores impossibles à valider. Après correction et reconstruction de l'image Docker, tous les recalculs de confiance se sont exécutés sans erreur.
